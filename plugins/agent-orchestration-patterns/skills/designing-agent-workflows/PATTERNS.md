# パターンと実例

7つのジャンル別パターン。**パートA**はClaude Code標準のサブエージェント（Task/Agent）ツールを使う——あなたがオーケストレーターとなり、ターンをまたいでエージェントを起動し、結果を統合する。**パートB**は、決定論的なワークフロー・スクリプティングツールを持つ環境向けの上級付録。

すべてのパターンは SKILL.md の「起動前チェックリスト」を前提とする。各パターンに**コピペ用プロンプト例・出力形式・失敗時方針・コスト目安**を添える。

## 共通：サブエージェント・プロンプト雛形

```
役割: <このエージェントは何者か>
コンテキスト（あなたは私の会話を見られない。必要なものはすべてここにある）:
  - ゴールと制約: <共有コントラクトのブロック>
  - 入力／パス: <ファイル、貼り付けた前段データ>
タスク: <1つの正確な仕事>
方法: <どうやるか — まずXを読む、既定で反証する 等>
正確に返すもの: <あとで解析する正確なフィールド／構造>
```

**コスト目安の読み方**：エージェント数 ≒ 起動数。`収集N + 検証(発見数×レンズ数) + 統合1`。同時実行には上限があり、超過分は波状実行になる。リスクが低ければレンズ数・ループ回数を削る。

# パートA — 標準のサブエージェント（Task/Agent）ツールで

あなたがオーケストレーターである。1回の「起動」は1つ以上のTask呼び出しに相当する。並行で走らせるには複数のTask呼び出しを**1つのメッセージ**に入れる。ある工程を前工程に依存させるには、**後続のメッセージ**で起動し、前の結果を新しいエージェントのプロンプトに貼り付ける。

## 1. レビュー／監査

**手順**：観点別レビュアーを並列 → 発見ごとに多レンズで検証 → 統合。

**コピペ用プロンプト（レビュアー）**
```
役割: あなたはTypeScriptのシニアレビュアー（観点=「正確性のバグ」専任）。
対象: <ファイルパス群> をまず読むこと。
タスク: 実行時/コンパイル時に実害を起こす本物のバグだけを挙げる。スタイル・好みは挙げない。
正確に返すもの: JSON配列。各要素 { file, line, severity(critical/high/medium/low), title, evidence(該当コード引用) }。
無ければ空配列。捏造禁止。
```
**コピペ用プロンプト（検証者・レンズ別／発見ごとに別レンズで複数）**
```
役割: あなたは独立した検証者（レンズ=「本当に発火するか」）。先入観を持たず反証を試みる。
対象: <file>:<line> を実際に読み、呼び出し元・型・ガードも追うこと。
検証する主張: <title> / 根拠: <evidence>
判定: 既定は refuted=true。実際に到達し実害が出る経路を示せたときのみ refuted=false。
正確に返すもの: { verdict: 立証済み|反証不能|要人間確認|棄却, refuted(bool), reason }
```
**採用ルール**：`立証済み` のみ採用。`反証不能` は保留（未確定として報告）。`棄却` は理由とともに記録。
**失敗時**：レビュアーが1観点落ちたら欠損を明示して継続。検証者が落ちた発見は「未検証」として保留。
**コスト**：観点数 + 発見数×レンズ数(2〜3) + 統合1。

## 2. 研究／エビデンス

**手順**：角度別に並列収集 → 主張ごとに独立検証（反証スタンス）→ 検証済みのみで起草。**棄却も残す**。

**コピペ用プロンプト（収集）**
```
役割: あなたは競合/事実調査のリサーチャー（担当=<対象/角度>）。
タスク: WebSearch/WebFetchで一次寄り情報を確認し、事実主張を3〜6件、各々に出典を付けて収集。
制約: 出典を出せない主張は出さない（捏造厳禁）。数値は出典の年度/時点を明記。
正確に返すもの: JSON配列。各要素 { statement(結論先行・1文・〜120字), citation(URL/名), sourceDate }
```
**コピペ用プロンプト（検証・1主張1体）**
```
役割: あなたは独立した事実検証者。反証するつもりで検証する。
タスク: 出典を実際に開き、主張を裏付けるか・時点が古くないか・誇張がないか確認。
主張: <statement> / 出典: <citation> / 時点: <sourceDate>
正確に返すもの: { isReal(bool), confidence(high/medium/low), correctedCitation(誤りなら修正版/なければ""), note }
```
**採用ルール**：`isReal=true かつ confidence≠low` を採用。`correctedCitation` があれば差し替え。**棄却した主張と理由を最終出力に残す**。
**失敗時**：検証が落ちた主張は採用しない（保留）。
**コスト**：角度数 + 主張総数×1（高リスクなら×複数レンズ）+ 起草1。

→ パートA末尾の**ケーススタディ**に実行記録（検証/棄却テーブル付き）あり。

## 3. 理解 → 設計 → 統合

**手順**：ドメイン別に並列理解（濃縮ダイジェスト）→ 設計者が並列でファイル書き出し → 1体が読み直して統合。

**コピペ用プロンプト（理解）**
```
役割: あなたは<ドメイン=要件/UX/backend等>の調査担当。対象資産をまず読む。
正確に返すもの: { summary(設計者がこれだけ読めば設計できる密度・〜2000字), key_facts[], gaps_or_risks[] }
```
**コピペ用プロンプト（設計）**
```
役割: あなたは<領域>の設計者。
コンテキスト: 下記は検証済みの調査ダイジェスト（再調査不要・正確な情報）:
<全ドメインのsummary/factsを貼る>
タスク: As-Is/To-Beのギャップ、MVPライン/拡張ライン、検証可能な完了基準つきロードマップを設計し、
本文は docs/design/<領域>.md に書き出すこと（mkdir -p）。
正確に返すもの: { file_path, summary(〜800字), open_questions[] }
```
**統合**：1体が各 file_path を読み直し、矛盾を指摘して1つの計画に統合。`open_questions`/`gaps` を「要人間判断」リストに集約。
**失敗時**：理解の一部欠損は明示して継続。設計失敗は当該領域をプレースホルダ化。
**コスト**：ドメイン数 + 領域数 + 統合1。

## 4. 変換／要約

**手順**：意味でグループ化 → グループごとに変換（**要素数を保存**）→ 重複しないキーで辞書マージ。

**コピペ用プロンプト**
```
役割: あなたは要約担当（担当キー=<グループ>）。共有入力 <パス> を読み、担当キーだけ処理。
厳守: 出力要素数＝入力要素数（削除・統合・追加なし）。事実を変えない。
保持: 最重要の数値・固有名詞・出典年（例: 売上高1.2兆円/2023年度/従業員900名）。複数あれば重要な1〜2個。
形式: 各項目を結論先頭・70字以内の1文に圧縮（スライド可読性のため）。
正確に返すもの: { result: { "<キー>": {...} } } を担当キーのみ
```
**統合**：各エージェントの `result` を1つの辞書に統合（キーが重ならない前提）。
**失敗時**：落ちたグループは再試行（要素数不一致も再試行対象）。
**コスト**：グループ数（1項目1体にしない）。

## 5. 移行／広域の機械的変更

**手順**：読み取り専用の調査 → 実装（番号付きコントラクト）→ 2トラック検証。

**コピペ用プロンプト（調査・読み取り専用）**
```
役割: あなたは移行の調査担当。Read/Grepのみ・変更禁止。
既知の出発点: <ファイル列>
調べること（番号付き）: 1.影響ファイル 2.正確なデータフロー 3.Xが未設定だとthrowする経路の有無 …
正確に返すもの: { target_paths[], flow_notes, throw_risks[] }
```
**コピペ用プロンプト（実装）**
```
役割: あなたは移行の実装担当。TDD。担当ファイル<排他リスト>以外は変更しない。
コンテキスト: 下記は調査結果（正確な情報・再探索するな）: <surveyのJSON>
コントラクト: 冪等な書き込み／入力検証（サイズ上限・実在チェック）／モジュール境界を越えない。
正確に返すもの: { files_changed[], summary, contract_deviations[], risks[] }
```
**検証（2トラック・並列）**：①実差分(`git diff`)を読む敵対的監査 `{ verdict, safe(bool), issues[] }` ②機械的グリーン化「ビルド/テストが通るまで修正、最大N回。テストが古ければ更新、コードが誤りなら修正」。
**失敗時**：実装の必須部が失敗→停止。並列編集は排他ファイルリストで衝突回避、共有ファイルは直列。
**コスト**：調査1 + 実装(分割数) + 監査1 + グリーン化1。

## 6. 検証専用（本番前ゲート）

実装した者とは独立した**別パス**として走らせる。

**コピペ用プロンプト（監査・複数を並列／最重要を先頭）**
```
役割: あなたは本番適用前の監査者（観点=<最重要リスク>）。実装者の報告は読まない。
対象: `git diff <対象>` を自分で読み、チェック項目を1つずつ突き合わせる。
項目: 1... 2... N...（机上トレース）。問題を見つけたら直接修正してよい。
正確に返すもの: { issues[]{severity(enum), title, fixed(bool), detail}, verdict(approve|approve_with_fixes|needs_work) }
```
**最終チェック**：ビルド＋テストをグリーンに。既知の偽陽性とその裁定ルール（テストを直すか実装を直すか）を事前に明示。1体の監査が落ちても報告の上で継続。
**コスト**：監査観点数 + 最終1。

## 7. 法務／法令検証（一次情報主義）

労働・社会保険法令やコンプライアンスなど、**誤った法的断定のコストが高い**領域の検証。研究／エビデンス（パターン2）の特化版で、出典を**法令の条文・行政通達の原文**に固定し、反証スタンスに加えて「一次情報と二次情報の峻別」「委任先の追跡」を組み込む。要点は、条文・通達の原文（一次情報）で裏を取り、解説記事や検索結果（二次情報）と峻別すること。

**手順**：論点分解（仮説の条文を特定）→ 並列で原文取得（法令原文取得ツール＋Web検索で関連通達・判例名を特定）→ 主張ごとに独立検証（原文に当たり反証）→ 一次情報を主軸に起草し、二次情報に依拠する箇所は明示。

**前提（共有コントラクトに入れる）**
- 条文・通達に言及するときは必ず**原文取得ツールで取得**し、「」で引用して出典URLを明記する。記憶だけで条文番号や通達内容を述べない。
- **一次情報**（原文取得ツールの結果）と**二次情報**（Web検索・Webフェッチ）を必ず区別して表示する。両者が矛盾するときは一次情報を正とする。
- 条文中の「政令で定める」「厚生労働省令で定める」等の**委任先**まで辿る。
- 告示・指針・判例は原文取得ツールの対象外のことがある → Web検索で補完する。官報・裁判所等の一次情報の原文に到達できた場合は出典URLを付して一次情報として扱い、解説記事等にとどまる場合は**二次情報として明示**（原文未確認の旨を注記）する。

**コピペ用プロンプト（原文取得）**
```
役割: あなたは労働・社会保険法令の調査担当。
ツール: 法令原文取得ツール（法令本文・行政通達を取得できるもの）と Web検索を併用。該当ツールが無い環境では、Web検索で e-Gov法令検索・官報など一次情報源の原文URLに到達して代替する。
タスク（番号付き・ラウンド制／最大4ラウンド）:
  1. 仮説の条文・通達を特定し、原文取得ツールで本文を取得
  2. Web検索で関連通達・判例の番号/名称を特定し、特定できた法令は原文取得ツールで本文取得
  3. 条文中の「政令で定める」「省令で定める」等の委任先も取得
終了条件（満たすまでループ）: 結論を支える条文を最低1つ原文引用／関係する行政通達を網羅的に検討／取得失敗を放置しない
正確に返すもの: JSON配列。各要素 { claim, primary[]{法令名, 条/項, 引用(「」原文), url}, secondary[]{要約, source, 信頼度(高/中/低)}, 委任先確認(bool), gaps[] }
制約: 原文を取得できない主張は「二次情報のみ・参考」と明示。捏造禁止。
```
**コピペ用プロンプト（検証・主張ごと）**
```
役割: あなたは独立した法令検証者。反証するつもりで検証する。収集者の結論は読まない（誘導防止）。
対象の主張: <claim> / 引用: <引用と url>
タスク: 引用された条文・通達の原文を自分で取得し直し、主張を裏付けるか／改正で失効していないか／委任先と矛盾しないか確認。
正確に返すもの: { verdict(立証済み|反証不能|要人間確認|棄却), refuted(bool), 原文引用(「」+url), reason }
```
**採用ルール**：`立証済み`（原文で裏取り済み）のみ採用。`反証不能`は保留。法解釈の当てはめや実務判断を要する論点は`要人間確認`としてエスカレーション（断定しない）。一次情報と二次情報を分けて提示し、二次情報のみに依拠する結論は「原文未確認のため参考」と注記する。
**失敗時**：原文取得が空振り → 別キーワード／目次から正しい条文番号を特定して再取得。2回空振りなら Web検索で通達名・番号を特定してから再試行。取得失敗を放置して結論を述べない。
**コスト**：論点数 + 主張総数×1（高リスクなら複数レンズ）+ 起草1。誤りのコストが高いほど検証を厚くする。

> **注意**：本パターンは*検証プロセス*の設計であって法的助言ではない。最終的な法令解釈・適用は有資格者の確認を要する。

## ケーススタディ（実行済みの研究パイプライン）

パターン2（研究／エビデンス）を一気通貫で実行した記録。

- **ゴール**：複数製品にわたる、出典付きの競合分析レポート。
- **構成**：収集（製品ごと1体）→ 検証（主張ごと1体・反証スタンス）→ 起草（検証済みのみ・ファイル書き出し）。
- **規模**：約30体規模のサブエージェント・数十万〜百万トークン規模・10分未満。

**検証結果テーブル（要約）**

| 段 | 件数 | 備考 |
|----|------|------|
| 収集した主張 | 数十件規模 | 複数製品 × 各観点（料金/機能/電子申請/運用/最新動向）|
| 採用（立証済み） | 大多数 | 出典を開いて裏取り済み |
| 棄却 | 少数（下記の1件など） | 下記 |

**棄却例（これが教訓）**

| 項目 | 内容 |
|------|------|
| 主張 | ある従量課金の課金開始人数を、実際より少ない人数と要約していた |
| 検証者の判定 | `isReal: false` / `confidence: medium` |
| 根拠 | 公式料金ページを複数回フェッチして原典を再確認。真の課金開始閾値はより多い人数だったと判明。一括りの条件設定が不正確 |
| 自己申告 | 「出典の要約処理が不確実性を生んだため、最終確定には人手確認推奨」と note に明記し断定を回避 |

反証スタンスの検証＋足切り＋訂正＋棄却保全が設計通り機能し、もっともらしいが誤った主張をレポート掲載前に捕捉した。**要点：主張ごとの検証パスこそが「速いが当てにならない」を「出典付きで信頼できる」に変えた。** 誤りのコストが高い場合は、主張ごとに複数レンズへ増やす。

# パートB — 付録：決定論的ワークフロー・スクリプティングツール（上級）

一部の環境には、サブエージェントをオーケストレーションするスクリプトを決定論的に実行するツール（出力スキーマを強制できるファンアウト／パイプラインAPI）がある。あれば、同じパターンがコードに圧縮できる。**そうしたツールが無いなら、この付録は無視してよい——パートAだけで完結している。**

そうしたツールでの正しさの要点：

- **すべての非同期オーケストレーション呼び出しを await する**（致命的かつ無音のバグ）。唯一の意図的な例外は、独立した裏方作業を**並走**させるためにPromiseを保持し、合流点で後から await する場合。
- **既定はパイプライン、バリアは「全前段結果を一度に必要とする工程」のみ。** 「scan → verify」を「全scan後に全verify」に分けない——検証を項目ごとのパイプラインに畳み、項目Aがまだscan中に項目Bを検証する。
- **出力スキーマを強制**し、手作業で解析しない。
- ツールが実行をジャーナル化して再開する場合、**非決定論的呼び出し**（時刻・乱数）を避ける。

パイプライン形状の例（JS風のファンアウトAPIの擬似コード）：

```javascript
// レビュー／監査：各観点の発見が返り次第その場で検証
const results = await pipeline(
  DIMENSIONS,
  d => agent(d.prompt, { schema: FINDINGS }),                 // レビュー
  review => parallel((review?.findings ?? []).map(f => () =>  // 各発見を3レンズで検証
    parallel(LENSES.map(lens => () =>
      agent(refutePrompt(f, lens), { schema: VERDICT })))
      .then(votes => decide(votes))))                         // 立証済みのみ採用、反証不能は保留
)
const confirmed = results.flat(2).filter(Boolean)             // ここのバリアは正当：統合は全発見を要する
```

こうしたツールの真の価値は決定論と再開であって、判断力ではない——判断は上記の設計原則に宿り、それはパートAと同一である。
