# サブエージェント・ワークフローの設計

## 概要

複数のサブエージェントを起動するのは簡単だ。だが**狙った品質の成果**を安定して引き出すのは難しい。その差はほとんどの場合モデルの地力ではなく、**オーケストレーションの設計**——作業をどう分割し、どう検証し、段から段へ結果をどう渡すか——にある。

このスキルはその設計のための、ツール非依存のプレイブックである。**標準のサブエージェント（Task/Agent）ツール**で起動する場合でも、環境に**決定論的なスクリプティングツール**がある場合でも、同じように適用できる。起動する前に読み、最後に「起動前チェックリスト」を回すこと。

**核心：品質を決めるのはオーケストレーションの設計である。** モデルの性能が上限を決めるが、その上限に届くか・設計不備で大きく下振れするかは設計次第だ。規律ある設計は下振れを防ぎ、規律のない設計は最上位モデルでも凡庸な結果を出す。

## 使うべきとき

- 1つのゴールのために複数のサブエージェントを起動する前（レビュー・監査・調査・設計・移行・要約・広域探索）
- 2つ以上のエージェントの結果を統合するとき
- 特にゴールに「網羅的」「徹底的」「正確に」「高確度」が含まれるとき

使わない場面：集約を伴わない、自己完結した単一サブエージェントの作業。

## 思考モデル：並走・検証・受け渡し

あらゆるファンアウトの品質は、次の3つの判断で決まる。

1. **並走 vs 同期。** 独立した作業は**並行**で走らせる。すべての前段結果を一度に必要とする工程の直前でのみ同期（全件待ち合わせ）する。習慣で同期すると壁時計時間を浪費し、ユーザーの時間を奪う。
2. **独立した証拠で検証する。** エージェントの自信ある回答は仮説であって結果ではない。確度は、別のエージェントが**成果物そのもの**を検証して初めて得られる——最初のエージェントの主張を読み返しても確度は上がらない。
3. **権威ある受け渡し。** 依存する段には、前段の構造化出力を「正確な情報」として渡す。そうすれば既知のことを再探索しない。

以下はすべて、この3点の実務的な形である。

## 設計の早見表（ゴール種別 → 推奨構成）

起動前に、ゴールがどの行かを決める。

| ゴール種別 | 構成 | 検証の深さ | カバレッジ戦略 |
|-----------|------|-----------|--------------|
| 軽い確認・要約 | 単発〜少数並列 | 単一票で可 | 単発パス |
| レビュー・監査 | 観点別に並列 → 発見ごと検証 | 発見ごと多レンズ（2〜3） | 観点を尽くす |
| 高確度が必要（バグ・事実・法務） | 並列収集 → 反証パネル | 反証N体・多数決＋立証区分 | 完全性クリティック |
| 網羅・徹底 | 収集 → 検証を繰り返す | 同上 | 新規が尽きるまでループ |
| 設計・計画 | 理解（並列）→ 設計（並列）→ 統合 | 統合段で矛盾検査 | open_questions を集約 |
| 移行・機械的変更 | 調査 → 実装 → 2トラック検証 | 監査＋ビルド/テスト | 影響範囲を調査で確定 |

迷ったら、リスク（誤りのコスト）が高いほど検証を厚く、低いほど薄くする。

## 6つの失敗モード（潰さなければ凡庸な結果になる）

これらはオーケストレーターが優秀なモデルでも発生する。各項目を、意識して防がない限り自分もやる、と仮定せよ。

### 1. 独立作業の直列化
何も共有しない作業を順番に走らせること。3つのエージェントが3つの別ファイルを読むなら、**まとめて**起動する（1メッセージに複数のTask呼び出し、または1つの並列ステップ）——順番にではなく。真に前段の結果に依存する作業だけを直列にする。逆に、**同じ成果物を変更する作業は並列にしない**（状態が壊れる）。それらは直列に。

### 2. 単一票の検証 ＋ 「反証不能」を「立証済み」と混同
1体のエージェントが「正しそう」と言い、それを信じること。確度・品質を要するゴールでは**パネルで検証**する：独立した検証者をN体起動し、各自に「反証せよ」または「別レンズで確認せよ」（正確性／セキュリティ／本当に再現するか）と指示する。各検証者は「証明されない限り反証扱い（refuted）」を既定にする。

結果は二値ではなく**4分類**で扱う：

- **立証済み**：検証者が独立した証拠で裏を取れた → 採用
- **反証不能**：誰も反証できなかったが、積極的な裏付けもない → **採用しない。「未確定」として保留**（反証されないことは正しさの証明ではない）
- **要人間確認**：判断に人の文脈・権限が要る → エスカレーション
- **棄却**：反証された → 落とす（理由とともに記録）

「過半数が反証しなかったから採用」は誤り。**積極的に立証されたものだけを採用**し、反証不能は別枠で扱う。

### 3. 網羅ゴールの単発パス
ユーザーが網羅的・徹底的・余さず、と言ったなら、1回の探索では不十分だ。**新規が尽きるまでループ**する（新しいものが出なくなるまでファインダーを回す。デデュープは「残したもの」ではなく「これまで見たすべて」に対して行う）か、**完全性クリティック**を足す（「何が欠けているか——未読のファイル、未検証の主張、試していない角度はないか」だけを問う最終エージェント）。その回答を次のラウンドへ送る。

### 4. サイレントな件数上限
結果を上限で切る（top-N、「最初の10ファイル」、サンプリング、リトライなし）のに、それを言わないこと。黙った上限は「全部見た」と誤読される。**すべての上限を**出力にもユーザーにも明示する。

### 5. 曖昧な受け渡し／再探索
下流エージェントに発見そのものを渡さず「発見を使え」と指示し、ゼロから再探索させること——遅く、ブレる。代わりに**前段の構造化結果を「正確な情報源」として次のプロンプトに注入する**（「これは検証済みのデータだ。再調査するな：…」）。

### 6. 実装者を信用した検証
作業をしたエージェント自身に良し悪しを尋ねる、あるいはその自己申告を事実として検証者に渡すこと。本物の検証工程は**成果物そのもの**（ファイル・差分・ソース）を、誰が作ったかに関係なく読む。本番前チェックは、成果物から状態を再構成する**独立した別パス**として走らせる。

## 見落とされがちな設計論点

失敗モードの回避に加え、ここを詰めると一段上がる。

- **相関した誤り（合議の限界）。** N体に同じ前提・同じ資料・同じプロンプトを渡すと、全員が同じように間違える。多数決は**誤りが独立しているとき**にしか効かない。検証者ごとにレンズ・資料・着眼点を変えて相関を断つ。全員一致は「正しい」ではなく「同じ盲点を共有している」かもしれないと疑う。
- **コンテキスト設計（何を渡し、何を渡さないか）。** 各エージェントに必要十分だけを渡す。多すぎると埋没・コスト増、少なすぎると的外れ。前段の生出力を丸投げせず、ダイジェストに圧縮して渡す。検証者には「検証対象」だけを渡し、収集者の自信や結論は渡さない（誘導を防ぐ）。
- **情報汚染。** あるエージェントの仮説や結論を他のエージェントに見せると、独立性が壊れ判断が偏る。並列で意見が欲しいときは、相互に結果を見せない。
- **失敗時リカバリ。** サブエージェントは失敗・空返し・タイムアウトしうる。各段に方針を決めておく：必須段の失敗は**即停止**（基盤がなければ下流は無意味）、並列の一部失敗は**欠損を明示して継続**（他の結果に価値がある）、一時的失敗は**回数上限つき再試行**。どれを採るかをゴールのリスクで決める。必要なツール・データ・権限が欠けているときは黙って劣化させず、何が欠けているかを明示する。
- **コスト・レイテンシ・品質のトレードオフ。** エージェント数・検証票数・ループ回数を増やすほど品質は上がるが、コストとレイテンシも増え、やがて逓減する。リスクが低い作業に5票の反証パネルは過剰。リスクが高い作業に単一票は過少。並列実行には同時実行数の上限があるため、大量ファンアウトは波状に実行される点も見込む。**規模はゴールの重大性に合わせる。** 目安：単一事実・低リスク=1票／レビュー・中リスク=2〜3レンズ／事実・法務・本番前=反証3〜5票＋新規が尽きるまでループ。手分けが十数体を超え、厳密な再現・再開が要る規模なら、標準のTask/Agent（パートA）から決定論ワークフロー（パートB）へ切り替える。

## 天井を上げるテクニック

- **自己完結プロンプト。** サブエージェントは会話を見られない。プロンプトにすべてを入れる：ファイルパス、正確な判定基準、前段データ、返してほしい正確な出力形式。サブエージェント品質への最大のレバー。
- **戻り値の形を指定する。** 確実に集約できるよう、各エージェントに返すフィールド／構造を正確に伝える（例：「リストで返せ。各要素は file, line, severity, evidence を持つ」）。ツールがスキーマを強制できるなら使う。できないなら形式を明示して検査する。
- **共有コントラクト定数。** ゴール・制約・禁止事項を一度書き、同じブロックを全エージェントのプロンプトに貼る。並列エージェントが単一の真実を共有し、ぶれない。
- **必須の自己申告フィールド。** `deviations`（逸脱）・`risks`・`open_questions` を返させる。どこで指示から外れたか、何が人間の判断を要するかが浮上する。
- **棄却した集合も残す。** フィルタする（採用 vs 棄却）ときは、棄却した項目と理由を報告する（監査可能性）。黙って捨てない。
- **採点平均より「足切り＋訂正」。** 事実・出典の作業では、二値の合否ゲート＋自動訂正版の方が、数値スコアより有効。
- **2トラック検証。** 「意味的に正しい／安全か？」（verdictを返す敵対的監査）と「機械的にグリーンか？」（ビルド・テスト・リンタが真偽を返す）を分ける。別エージェント・別レンズ。
- **意味でグループ化。** 1項目1エージェントにせず、関連項目を少数のグループに束ねる。過剰ファンアウトとコンテキスト重複を防ぐ。
- **大きな入力は参照で渡す。** 共有する大入力はファイルに書き、各エージェントに「読んで自分の担当範囲だけ処理せよ」と伝える。全プロンプトに貼り付けない。
- **終了条件＋上限回数。** 長時間／修正系の工程には、検査可能な成功条件**と**最大反復回数を与える（「ビルドが通るまで修正、最大8回」）。さらに収束させる裁定ルール（「テストが古いならテストを更新、コードが誤りならコードを修正」）。
- **要素数の保存。** 要約・翻訳・整形では「出力要素数＝入力要素数。削除・統合・追加をしない。事実を変えない」と明示する。

## 起動前チェックリスト（起動する前に回す）

- [ ] **早見表で構成を決めた**：ゴール種別 → 構成・検証深さ・カバレッジ
- [ ] **並走**：独立エージェントは並行起動。真に全件必要な工程だけ同期。同一成果物の変更は直列
- [ ] **検証の深さがゴールに一致**：確度・品質ゴール → 多レンズ／反証パネル。採用は「立証済み」のみ、「反証不能」は保留
- [ ] **誤りの独立性**：検証者ごとにレンズ・資料を変え、相関した誤り（全員同じ盲点）を断つ
- [ ] **カバレッジがゴールに一致**：網羅ゴール → 新規が尽きるまでループ or 完全性クリティック
- [ ] **自己完結プロンプト**：各プロンプトにパス・基準・前段データ・必要な戻り値形式を内包
- [ ] **戻り値の形を指定**（可能ならスキーマ強制）。結果は前提せず検査する
- [ ] **権威ある受け渡し**：依存段は前段結果を「正確な情報」として受け取る。再探索なし
- [ ] **検証の独立性**：チェックは成果物（ファイル／差分／ソース）を読む。作成者の自己申告を読まない
- [ ] **コンテキスト設計**：各エージェントに必要十分だけ渡す。検証者に収集者の結論を見せない
- [ ] **失敗時リカバリ**：必須段=即停止／並列一部失敗=欠損明示で継続／一時失敗=回数上限つき再試行
- [ ] **自己申告＋棄却の保全**：必要箇所で deviations／risks／open_questions を要求。フィルタ済み項目は報告（破棄しない）
- [ ] **サイレント上限なし**：top-N／サンプリング／リトライなし等の上限をすべて明示
- [ ] **終了条件**：長時間／修正系は成功条件＋最大回数。裁定ルールを固定してループを収束させる
- [ ] **規模が重大性に一致**：エージェント数・検証票数・ループ回数をリスクに見合わせ、過剰・過少を避ける
- [ ] **要素数**（変換・要約）：「出力数＝入力数、削除・統合・追加なし、事実改変なし」を明示

## パターンと実例

7ジャンル（レビュー／監査・研究・設計・変換・移行・検証専用・法務検証）のすぐ使えるパターンを、コピペ可能なプロンプト例・出力スキーマ例・失敗時方針・コスト目安とともに **PATTERNS.md** に収録。標準のサブエージェント（Task/Agent）ツール向けに記述し、決定論的スクリプティングツールを持つ環境向けの上級付録を添えてある。研究パイプラインを実際に完走させたケーススタディ（約30体規模のエージェント・数十万〜百万トークン規模・数十件規模の主張を収集→大多数を採用・少数を棄却）も、検証テーブル・棄却例つきで実例として記載。

## よくある間違い

| 間違い | 直し方 |
|--------|--------|
| エージェントを1体ずつ順に起動 | 独立なものはまとめて起動 |
| 「これをレビューして良し悪しを教えて」（1体） | 反証者／レンズのパネル。立証済みのみ採用 |
| 「誰も反証しなかった」を採用 | 反証不能は保留。積極的に立証されたものだけ採用 |
| N体に同じ資料・同じ問いを渡し全員一致を信じる | レンズ・資料を変え相関を断つ。全員一致=共有盲点を疑う |
| 「徹底的に」を1回の探索で | 新規が尽きるまでループ or 完全性クリティック |
| 「さっきの発見を使って」（中身を渡さない） | 発見を正確な情報として注入 |
| 作成者の「完了しました」を信じる | 成果物を独立に読んで検証 |
| 黙って10件で打ち切り | 上限を明示 |
| 低リスク作業に5票パネル／高リスク作業に単一票 | 規模を重大性に合わせる |
| 共有コンテキストを前提にしたプロンプト | すべてのプロンプトを自己完結に |
